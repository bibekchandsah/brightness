Fixed. The root cause was that the old code tracked pressed keys in a set() and removed them on release — but over time, release events get missed (screen lock, Alt+Tab, sleep/wake, losing focus). The set fills up with "ghost" keys that never get removed, so the detected combo includes stale extra keys and stops matching.

What changed: Instead of maintaining our own key state, the app now uses ctypes.windll.user32.GetAsyncKeyState() to query modifier keys (Ctrl, Shift, Alt) directly from Windows on every keypress. This is always accurate — no state tracking, no stale keys, works reliably even after hours of uptime.

Tray menu — a checkable "Start on Startup" item (with separators) that reflects the current state on launch and toggles it when clicked

Tray menu additions
"Check for Updates" — manual check, shows a dialog if already up-to-date or if the check fails
"Update Available: vX.Y.Z" — hidden until an update is detected; clicking it triggers the install flow

Install flow
Compiled exe: downloads the asset → writes a temporary .bat that waits 2 s, replaces the exe with move /y, then relaunches it → quits the current process
Dev/Python mode (or no .exe asset in release): opens the GitHub releases page in the browser instead
